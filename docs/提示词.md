### 模块目标
这是一份面向 AI 的 FocusGS 项目知识底稿。它的主要用途不是给人直接阅读，而是作为后续生成终期文档、项目介绍、答辩材料、阶段总结时的统一事实来源。

### 你需要知道的事实
- 项目名称：`FocusGS`
- GitHub 仓库：`https://github.com/Shirowana/FocusGS.git`
- 这份文档服务于“终期文档生成”，不是泛用产品手册。

### 建议重点强调
- FocusGS 不是单纯的算法复现，也不是单纯的网页展示，而是一条从三维重建到轻量化发布的完整链路。
- 项目有两条主线：
  1. 基于 `MEGS²` 思路的显存优化重建
  2. 训练结果的轻量化保存、格式转换与网页展示（注意这里的轻量化保存也是一种模型算法，用一些手段比如剪枝（移除低贡献gaussian）、量化（属性量化编码，降低储存精度）、格式转换（转换为web友好的紧凑格式）把原来1G+的结果文件变成只有100-200MB，非常轻量，你可以参考LightGaussian、Supersplat这之类的工作，但是我们不是完全照抄这样的，而是大致按照剪枝、量化、格式转换这个思路来做的。所以你在这里可以基于这个思路自由发挥）

### 不要写错的地方
- 不要把当前仓库的历史演进过程和最终成品设想混为一谈。
- 不要把未验证的数据写成确定事实。

## 项目全局概览

### 模块目标
让 AI 先快速理解 FocusGS 到底在做什么、要解决什么问题、与现有方法相比站在什么位置。

### 你需要知道的事实
- FocusGS 面向的是多视图图像驱动的三维重建与展示任务。
- 项目希望把三维重建流程做成一条更完整、更易展示、更适合有限硬件条件的链路。
- 相比传统只关注重建质量的系统，FocusGS 同时关注：
  - 重建质量
  - 训练与渲染显存占用
  - 结果文件体积
  - 网页端加载与预览成本
- FocusGS 采用 `3D Gaussian Splatting` 作为三维重建与渲染主线。
- 在此基础上，项目引入 `MEGS²` 的核心思想，用于显存优化与表示压缩。
- 除了训练本身，FocusGS 还关注训练后结果如何更轻量地保存、导出和发布。

### 建议重点强调
- 一句话定义可以写成：
  `FocusGS 是一个围绕 3D Gaussian Splatting 构建的轻量化三维重建与网页展示平台，强调从显存优化训练到轻量化结果发布的完整链路。`
- 项目最终目标不是只“训出一个模型”，而是让重建结果更容易进入网页展示、工作台管理和后续复用。

### 不要写错的地方
- 不要把 FocusGS 写成纯 NeRF 项目。
- 不要把它写成只做前端展示的网站。
- 不要忽略“轻量化结果保存”这条主线。

## 最终成品形态说明

### 模块目标
让 AI 知道最终成品长什么样，用户从哪里进入系统、如何操作、系统如何把算法能力包装成可展示的产品。

### 你需要知道的事实
- 项目由首页展示区与工作台两大部分组成。
- 首页负责叙事、方法说明、场景预览、工作流展示与项目导览。
- 工作台负责输入数据、配置参数、发起训练、查看日志、管理历史任务、断点续训和结果导出展示。

  1. 浏览首页了解方法与场景

  8. 在历史记录中查看任务、失败状态或断点续训入口

## 项目目录与代码组织知识包

### 模块目标
让 AI 或组员快速理解 `FocusGS` 仓库当前真正有用的结构，知道前端、算法、数据、运行产物分别在哪里，避免把已经废弃或仅作历史参考的目录误当成主线。

### 你需要知道的事实
- 当前 `FocusGS` 仓库中，真正需要优先理解的部分可以分成四类：
  1. 前端展示与工作台
  2. 算法训练主线
  3. 本地数据与模型资产
  4. 项目说明与未完成项文档
- 当前推荐 AI 理解的“主线目录”如下：

#### 一、前端与工作台主线

- `gs-browser-viewer/`
  当前项目最核心的前端目录，承担首页展示、场景浏览、工作台、训练日志展示、历史记录、断点续训、结果导出展示等功能。

- `gs-browser-viewer/src/main.js`
  前端入口文件，负责页面切换、首页/工作台渲染入口、Viewer 初始化与部分全局交互逻辑。

- `gs-browser-viewer/src/ui/renderApp.js`
  最核心的页面渲染文件。首页结构、工作台结构、时间线、参数卡片、历史记录、结果导出展示等大量界面逻辑都在这里组织。

- `gs-browser-viewer/src/config/loadScenes.js`
  读取场景元数据，决定前端有哪些场景、默认加载哪个场景。

- `gs-browser-viewer/src/config/loadHistory.js`
  读取历史记录数据，用于工作台的历史任务视图。

- `gs-browser-viewer/src/viewer/`
  浏览器端 3DGS Viewer 相关逻辑目录。
  其中：
  - `createViewer.js` 负责创建 viewer
  - `loadScene.js` 负责加载静态场景
  - `loadEmbeddedViewer.js` 负责嵌入式预览逻辑

- `gs-browser-viewer/styles.css`
  全局样式文件。首页视觉风格、深浅色主题、卡片样式、方法页、工作流页、工作台页等风格控制都集中在这里。

- `gs-browser-viewer/public/data/scenes.json`
  场景配置核心文件。定义每个场景的名称、路径、简介、封面等，是场景展示链路的重要入口。

- `gs-browser-viewer/public/data/history.json`
  历史记录相关静态数据入口之一，用于工作台任务历史展示。

- `gs-browser-viewer/public/gallery/`
  首页场景导览区或场景相关静态素材目录。

- `gs-browser-viewer/public/thumbnails/`
  场景缩略图目录。

- `gs-browser-viewer/figures/`
  首页方法页、工作流页、场景视频、结果导出展示等所需的静态图片和视频素材目录。很多首页展示资源都来自这里。

- `gs-browser-viewer/server/localApi.js`
  当前本地工作台最关键的接口层。它不是云端后端，而是本地开发环境下的 Node 侧训练桥接逻辑，负责：
  - 创建任务目录
  - 解析输入模式
  - 定位本地数据集路径
  - 组织训练命令
  - 记录日志、状态、checkpoint 等运行信息

- `gs-browser-viewer/package.json`
  前端项目依赖与脚本入口。当前本地启动端口是 `4173`。

#### 二、算法与训练主线

- `MEGS-2/`
  当前算法主线最重要的目录。FocusGS 在终期文档中的显存优化重建部分，应主要围绕这个目录对应的方法来写。

- `MEGS-2/train.py`

## 算法模块知识包

### 模块目标
清楚说明 FocusGS 在算法层到底借鉴了什么、改造了什么、创新点落在哪里，以及这部分和最终平台如何衔接。

### 你需要知道的事实
- FocusGS 的算法主线以 `3D Gaussian Splatting` 为基础。
- 之所以选择 3DGS，是因为它在新视角合成与实时显示之间取得了较好的平衡，且天然适合点/高斯形式的快速展示。
- FocusGS 在 3DGS 基础上引入 `MEGS²: Memory-Efficient Gaussian Splatting via Spherical Gaussians and Unified Pruning` 的思路。
- `MEGS²` 的关键思想包括：
  - 用更轻量的 `Spherical Gaussian` 颜色表示替代传统的 `Spherical Harmonics`
  - 使用统一剪枝策略 `Unified Pruning`
  - 同时压缩高斯 primitive 数量与单个 primitive 的参数量
  - 目标是在尽量保持重建质量的同时降低显存占用与运行负担
- FocusGS 对算法侧的承接重点有两条：
  1. 借助 `MEGS²` 思想构建更轻的重建主线
  2. 对训练结果继续进行轻量化保存与格式转换，形成适合网页传输与预览的输出
- 当前仓库中同时保留了：
  - `gaussian-splatting/`：历史上已打通的训练主线与浏览器展示资产来源
  - `MEGS-2/`：用于方法研究、参数参考和后续替换/扩展的核心算法快照
- 从项目叙事角度，终期文档中需要把 FocusGS 的算法创新表达成：
  - 第一层创新：重建阶段的显存优化（这里就是MEGS-2）
  - 第二层创新：结果阶段的轻量化保存与发布（这里不是MEGS-2，而是一种剪枝、量化、格式转换的优化方法）

### 建议重点强调
- 不要只写“我们用了 MEGS²”，而要写“我们为什么引入它，以及它在 FocusGS 中承担什么作用”。
- 可以把算法部分概括为：
  `FocusGS 沿用 3DGS 的三维高斯表示主线，并结合 MEGS² 的轻量化表示与剪枝思路，降低训练与渲染的显存压力；同时面向网页展示场景，对训练结果进一步组织为更轻量的发布资产。`
- 要明确区分：
  - 论文方法借鉴/集成：Spherical Gaussian、Unified Pruning、显存优化思路
  - 项目系统侧创新：结果轻量化保存、导出组织、网页展示衔接、工作台交互包装

### 不要写错的地方
- 不要把 `MEGS²` 全部写成 FocusGS 原创论文。
- 不要把“结果轻量化保存”误写成 `MEGS²` 原论文内容。
- 不要忽略 FocusGS 是“算法 + 系统平台”的结合体。

### 待补充信息
- FocusGS 最终采用统一的 `MEGS²` 训练配置作为默认参数模板，在 `30K iterations` 主设置下完成多场景训练，并支持按场景类型切换室内/室外的重要度度量与图像尺度选择。
- 轻量化资产保存最终采用“剪枝 + 量化 + 格式转换”的组合策略：先移除低贡献 Gaussian，再对关键属性进行压缩编码，最后转换为适合网页传输与加载的紧凑结果格式。
- 最终版本保留 checkpoint 保存与断点续训能力，同时将最终展示重点放在可复用的最终结果与轻量化导出资产上。
- 算法流程图、结构图与伪代码均已整理完成，可围绕“输入数据 -> MEGS² 训练 -> 轻量化后处理 -> 网页发布资产”这条主线撰写。

## 算法文献阅读建议

### 模块目标
告诉算法部分的组员或 AI：如果要真正写懂 FocusGS 的算法章节，应该优先阅读哪些论文，以及这些论文分别解决什么问题。

### 你需要知道的事实
- 算法部分建议按“技术演进 + 项目直接相关性”的顺序阅读。
- 推荐至少阅读下面几类文献：

#### 一、NeRF 起点

- `NeRF: Representing Scenes as Neural Radiance Fields for View Synthesis`
  - arXiv: https://arxiv.org/abs/2003.08934
  - Project page: https://www.matthewtancik.com/nerf
  - 建议用途：理解神经辐射场为什么成为后续新视角合成的重要基础，以及 3DGS 为什么被视为对 NeRF 路线的重要加速和替代。

#### 二、MVSNet 路线

- `MVSNet: Depth Inference for Unstructured Multi-view Stereo`
  - arXiv: https://arxiv.org/abs/1804.02505
  - 建议用途：作为深度学习三维重建早期主线代表方法，帮助梳理“从多视图立体匹配到神经表示”的技术演进。

#### 三、3D Gaussian Splatting 主线

- `3D Gaussian Splatting for Real-Time Radiance Field Rendering`
  - arXiv: https://arxiv.org/abs/2308.04079
  - Official project page: https://repo-sam.inria.fr/fungraph/3d-gaussian-splatting/
  - Official GitHub: https://github.com/graphdeco-inria/gaussian-splatting
  - 建议用途：这是 FocusGS 的最核心基础论文。需要读懂它的三维高斯表示、优化流程、实时渲染优势，以及为什么它适合浏览器展示与工程落地。

#### 四、MEGS² 主线

- `MEGS²: Memory-Efficient Gaussian Splatting via Spherical Gaussians and Unified Pruning`
  - arXiv: https://arxiv.org/abs/2509.07021
  - Project page: https://megs-2.github.io
  - Official GitHub: https://github.com/IGL-HKUST/MEGS-2
  - 建议用途：这是 FocusGS 显存优化重建部分的直接方法来源。需要重点理解：
    - 为什么用 `Spherical Gaussian` 替代传统 `SH`
